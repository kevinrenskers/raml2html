HTTP standards

HTTP Verbs

VerbDescription
GETUsed for retrieving resources
POSTUsed for creating resources
PATCHUsed for updating resources with partial JSON data. For instance, an Issue resource has title and body attributes. A PATCH request may accept one or more of the attributes to update the resource. PATCH is a relatively new and uncommon HTTP verb, so resource endpoints also accept POST requests
PUTUsed for replacing resources or collections. For PUT requests with no body attribute, be sure to set the Content-Length header to zero
DELETEUsed for deleting resources

HTTP Success codes

CodeDescription
200OK. Valid API call and response
201Created. Resource was successfully created

HTTP Error codes

CodeDescription
400Bad Request. Sending invalid JSON or invlaid JSON values
401Unauthorized. Requesting a Resource without permission
404Not Found. The requested resource could not be found but may be available again in the future
422Unprocessable Entity. Sending invalid JSON fields
500Bad Code. We forgot to squash a bug. Sorry :(
503Service Unavailable. This service is temporarily unavailable
504Gateway Timeout. A back-end service did not provide a timely response

HTTP Redirects

CodeDescription
301Permanent redirection. The URI you used to make the request has been superseded by the one specified in the Location header field. This and all future requests to this resource should be directed to the new URI
302, 307Temporary redirection. The request should be repeated verbatim to the URI specified in the Location header field but clients should continue to use the original URI for future requests

Rate Limits

You can make up to 500 requests per 15 minutes for the 'core' service. Additional service limits may be introduced over time. These limits may be increased for your account by agreement.

Going over your limit

Once you go over the rate limit for a service you will receive an error response.

GET /clients

HTTP/1.1 403 Forbidden

{
"errors": [
  {
    "message": "You have exceeded the rate limit for the \"core\" service.\nYour allowance is 500 per 15 minutes.\nThis allowance will reset in 35 seconds.",
    "name": "RateLimitError",
    "docs": ".../api/overview/rate-limits"
    "status": 403,
  }
]
}

Get your current rate limit statuses

You can check your current rate limit statuses at any time without affecting your rate limits.

For future proofing the rate limit is broken down by category. This allows us to set different limits by resource (Learning lessons from GitHubs api). The "core" object provides your rate limit status for all current APIs.

NameDescription
concurrentAllowanceThe maximum number of requests permitted at the same time.
allowanceThe maximum number of requests permitted per duration.
minutesDuration that the allowance applies for.
remainingThe number of requests remaining in the current rate limit window.
resetThe time in seconds until the current rate limit window resets.

GET /rate-limits

HTTP/1.1 200 OK

{
"data": {
  "core": {
    "concurrentAllowance": 6,
    "allowance": 500,
    "remaining": 324,
    "reset": 120
  }
}
}
